## 1) DATE

### Визначення
`DATE` — це календарна дата в тексті, переважно в числовому українському форматі:
- `DD.MM.YYYY`
- `D.M.YYYY`
- також допускаються роздільники `-` та `/`, якщо вони трапляються

### Формат виходу (нормалізація)
- `value`: нормалізована дата у форматі `YYYY-MM-DD`, якщо дата валідна
- якщо парсинг невизначений/невалідний: зберігати `raw_date` і ставити `parsed_date=None`

### Що НЕ вважаємо DATE (негативні приклади)
- лише рік: `2016`
- номери законів/статей/проваджень, які схожі на числа
- десяткові числа: `3.14`
- нумерація списку: `1.`, `2.`, `3.`

### Edge cases із цього датасету
1. `... договору №3 від 25.05.2011 року ...` -> витягати тільки `25.05.2011`, не `№3`.
2. `... експертиза від 08.07.2014.` -> фінальна пунктуація не входить у span.
3. `... провадження № 12016100000000042 від 27.01.2016 року ...` -> коректно витягати і DATE, і DOC_ID без плутанини перекриттів.
4. `... від 31.10.2007 р.` -> короткий суфікс `р.` не має ламати матчинг.
5. `... від 10.04.2008 №610 ...` -> сусідній `№610` це контекст DOC_ID, а не частина дати.

## 2) AMOUNT

### Визначення
`AMOUNT` — грошова сума з явним маркером валюти поруч:
- форми UAH: `грн`, `гривень`, `гривня`, `гривні`, `₴`, `UAH`
- форми USD: `доларів`, `долари`, `USD`, `$`
- форми EUR: `євро`, `EUR`, `€`

### Формат виходу (нормалізація)
- `value`: числове значення `float` (наприклад, `1218.0`)
- `currency`: одне зі значень `UAH | USD | EUR | UNKNOWN`
- оригінальний фрагмент зберігати у `raw_value`

### Що НЕ вважаємо AMOUNT (негативні приклади)
- відсотки: `100%`
- числа без валюти: `500`
- роки: `2014`, `1997`
- нумерація: `№ 24`

### Edge cases із цього датасету
1. `... становить 1218 гривень ...` -> лексична форма UAH.
2. `... нещасні 800 гривень пенсії ...` -> прикметник перед сумою не заважає валідному витягу.
3. `... не більше 13 гривень.` -> кінцева пунктуація не входить у span.
4. `... маючи останні 500 доларів ...` -> лексична форма USD без символу.
5. `... візьме якихось 100 доларів за таксі ...` -> розмовний контекст теж має матчитися.

## 3) DOC_ID

### Визначення
`DOC_ID` — ідентифікатор документа/справи/наказу, переважно з маркером або контекстом:
- форма з маркером: `№ <value>`
- контекстні якорі поруч (приклади): `договір`, `провадження`, `постанова`, `розпорядження`, `справа`, `наказ`, `рахунок`

### Формат виходу (нормалізація)
- `type`: тип, виведений з локального контексту (`CONTRACT_ID`, `CASE_ID`, `ORDER_ID`, `GENERIC_DOC_ID`)
- `value`: очищене значення ідентифікатора (без `№` і зовнішньої пунктуації)
- зберігати `raw_value`

### Що НЕ вважаємо DOC_ID (негативні приклади)
- `№` як нумерація списку/випуску без документного контексту
- випадкові короткі числа без маркера/контексту
- посилання на статті/частини, що не є id документа

### Edge cases із цього датасету
1. `... від 2 жовтня 2015 року за № 859.` -> валідний generic id із маркером.
2. `... рішення ... від 24 грудня 2015 року № 58` -> id у кінці речення.
3. `... у таборі № 36 у Кучино ...` -> треба відхиляти як недокументний id (номер табору/локації).
4. `... провадження № 12016100000000042 ...` -> довгий numeric id зберігати без змін.
5. `... (Відомості ... 1997 р., № 24, ст. 170 ...)` -> часто номер випуску/посилання, зазвичай відхиляти без сильного документного контексту.
